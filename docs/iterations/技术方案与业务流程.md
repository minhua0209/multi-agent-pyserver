# 当前项目技术方案与业务流程

更新时间：2026-07-14

## 1. 项目定位

当前项目是一个 Python 3.12 + FastAPI 的 API-only MVP，用于验证“多智能体任务协作中心”的核心链路。

主流程不是简单地把请求拆成多个静态任务，而是收敛为一个主任务。主任务经人工确认后，由 LangGraph 编排进入多轮执行：每一轮读取当前上下文，规划子任务，执行子任务，将结果写回上下文，再判断是否需要下一轮。

模型调用优先走 OpenAI 兼容接口；模型不可用或返回异常时，核心流程会回退到 mock 逻辑，保证本地 demo 可以继续运行。

## 2. 技术方案

### 2.1 技术栈

- Python 3.12
- FastAPI：提供 HTTP API
- Pydantic v2：定义请求、响应和领域模型
- LangGraph：编排确认后的多轮任务执行流
- Uvicorn：本地开发服务
- pytest + httpx：测试 FastAPI API 和服务逻辑

依赖定义在 `pyproject.toml`。

### 2.2 模块结构

- `app/main.py`
  - 创建 FastAPI 应用。
  - 初始化 AgentRegistry、InMemoryTaskStore、TaskService。
  - 挂载 agent 和 task 两组路由。

- `app/api/`
  - `routes_agents.py`：agent 注册、agent 列表、agent 轮询已分配任务。
  - `routes_tasks.py`：任务请求创建、任务确认、结果提交、任务查询。

- `app/core/`
  - `models.py`：核心 Pydantic 模型。
  - `enums.py`：任务状态、节点状态、来源类型、结果状态。
  - `model_client.py`：模型调用、模型响应解析、意图识别、多轮分发、子任务执行协议。
  - `mock_llm.py`：模型失败时的本地 mock 回退逻辑。

- `app/services/`
  - `storage.py`：agent 本地 JSON 存储、task 内存存储。
  - `task_service.py`：任务请求、确认、结果提交、依赖释放等业务编排。
  - `tool_executor.py`：执行 agent 声明的工具。

- `app/workflows/`
  - `task_graph.py`：LangGraph 多轮任务状态机。

- `tests/`
  - 覆盖 agent 注册、任务 API、模型响应解析、LangGraph 流程、工具执行器。

### 2.3 核心数据模型

- `Agent`
  - 表示可处理任务的智能体。
  - 包含 `name`、`description`、`capabilities`、`tools`。

- `AgentTool`
  - 表示 agent 可用工具。
  - 包含 `name`、`description`、`type`、`config`、`input_schema`。
  - 当前可执行类型：`mock`、基础 `http`。

- `Task`
  - 主任务实体。
  - 包含来源内容、确认后的标题描述、当前状态、当前节点、上下文、最终输出、事件记录。

- `TaskDraft`
  - 意图识别后的草稿。
  - 包含标题、描述、置信度、建议处理方、建议 agent、兼容用依赖字段。

- `RoundPlan`
  - 分发 agent 每轮返回的执行计划。
  - 包含是否继续、本轮执行模式、分发原因、子任务列表、最终输出。

- `SubTask`
  - 每轮动态生成的执行单元。
  - 包含标题、描述、分配 agent、工具调用、工具结果、执行输出。

- `TaskContext`
  - 主任务持续累积的上下文。
  - 包含 `summary`、`rounds`、`artifacts`。

- `TaskRound`
  - 一轮执行记录。
  - 保存执行前上下文、子任务列表、执行后上下文。

### 2.4 存储方案

当前存储是 MVP 级别：

- Agent 使用本地 JSON 文件持久化。
  - 默认路径由 `app/main.py` 指向 `app/data/agents.json`。
  - 测试中可通过 `create_app(agent_file=...)` 注入临时文件。

- Task 使用内存字典存储。
  - 服务重启后任务会丢失。
  - 当前适合本地 demo 和流程验证，不适合作为生产持久化方案。

### 2.5 模型调用方案

模型客户端在 `app/core/model_client.py`。

主要能力：

- 意图识别：根据请求内容和已注册 agent 列表生成任务草稿。
- 多轮分发：根据主任务、当前上下文和可用 agent 返回下一轮子任务。
- 子任务执行：根据主任务上下文、子任务信息、agent 工具和工具结果生成输出。
- 完成判断：保留了完成度判断模型函数，但当前主流程主要依赖分发结果的 `should_continue` 判断结束。

模型响应要求返回 JSON。客户端会对 Markdown code block 和响应中的 JSON 片段做容错解析。

### 2.6 LangGraph 编排方案

确认后的主任务由 `TaskGraphRunner` 执行。

图节点：

1. `round_dispatch`
   - 设置当前节点为 `dispatch_decision`。
   - 调用模型规划下一轮。
   - 模型不可用时使用 `mock_round_plan`。
   - 如果有子任务，递增 `loop_count`。

2. `subtask_execution`
   - 设置当前节点为 `subtask_execution`。
   - 为每个子任务解析处理 agent。
   - 有 agent 则走 agent 执行链路；没有 agent 则走 human mock 节点。

3. `context_update`
   - 设置当前节点为 `context_update`。
   - 将本轮输出合并进 `TaskContext.summary`。
   - 追加一条 `TaskRound`。

4. `completion_judge`
   - 设置当前节点为 `completion_judge`。
   - 如果分发计划表示无需继续，任务置为 `succeeded`。
   - 否则回到 `round_dispatch`。

5. `human_intervention`
   - 超过最大轮数时进入。
   - 当前记录人工介入事件，但不自动完成任务。

最大循环轮数由 `Task.max_loop_count` 控制，当前默认是 10。

### 2.7 工具执行方案

工具执行器在 `app/services/tool_executor.py`。

执行链路：

1. 子任务执行模型第一次被调用。
2. 如果模型返回 `tool_calls`，系统查找 agent 注册过的工具。
3. 系统执行工具。
4. 工具结果写入 `subtask.tool_results`。
5. 子任务执行模型第二次被调用。
6. 模型基于 `tool_results` 生成最终输出。
7. 最终输出写入 `subtask.output` 和主任务上下文。

当前支持：

- `mock`
  - 直接返回工具配置里的 `response`。

- `http`
  - 支持基础 URL 模板替换。
  - GET、DELETE 不带 body。
  - 其他方法用工具调用参数作为 JSON body。

当前未覆盖：

- 鉴权配置
- 重试策略
- 工具调用审计
- 复杂参数校验
- 安全白名单

## 3. API 入口

### 3.1 Agent API

- `POST /api/v1/agents`
  - 注册 agent。
  - 可声明能力标签和工具列表。

- `GET /api/v1/agents`
  - 查询已注册 agent。

- `POST /api/v1/agents/{agent_id}/poll`
  - 查询当前分配给某个 agent 的任务。
  - 实现方式是从内存任务列表中过滤 `assigned_agent_id`。

### 3.2 Task API

- `POST /api/v1/tasks/requests`
  - 创建任务请求。
  - 返回一个 request id 和一个主任务。
  - 主任务进入 `human_confirmation`。

- `POST /api/v1/tasks/{task_id}/confirm`
  - 人工确认任务标题和描述。
  - 确认后自动进入 LangGraph 多轮执行。

- `POST /api/v1/tasks/{task_id}/result`
  - 提交外部执行结果。
  - 如果结果失败，任务置为 failed。
  - 如果 `should_complete=true`，任务置为 succeeded。
  - 否则继续跑自动流程。

- `GET /api/v1/tasks/{task_id}`
  - 查询单个任务。

- `GET /api/v1/tasks`
  - 查询全部内存任务。

## 4. 实际业务流程

### 4.1 注册处理 agent

业务方先注册可处理任务的 agent。

示例场景：

- 报价 agent：能力包含 `quote`、`crm`。
- 合同 agent：能力包含 `contract`、`risk`。
- CRM agent：能力包含 `crm`，并声明 `crm_query` 工具。

注册后的 agent 会被后续意图识别和任务分发使用。

### 4.2 上游创建任务请求

上游系统、人或其他 agent 调用 `POST /api/v1/tasks/requests`。

输入包含：

- `source_type`
- `content`
- `metadata`

系统读取当前 agent 列表，将请求内容和 agent 能力一起交给模型做意图识别。

### 4.3 意图识别生成主任务草稿

模型可以识别出一个或多个 draft。

当前实现会把多个 draft 合并为一个主任务：

- 标题使用多个 draft 标题拼接。
- 描述按列表合并。
- 置信度取最低值。
- 建议处理方取第一个 draft 的建议。

主任务创建后：

- `task_status = running`
- `current_node = human_confirmation`
- `context.summary = ""`
- `context.rounds = []`

### 4.4 人工确认主任务

人工确认是强制步骤。

确认时写入：

- `title`
- `description`

确认后系统检查兼容字段 `dependency_task_ids`。

- 如果依赖未完成，任务进入 `waiting_dependencies`。
- 如果无依赖或依赖已完成，进入自动执行流程。

当前主路径已切换为“一个主任务 + 多轮上下文调度”，依赖字段主要保留兼容。

### 4.5 第一轮分发

LangGraph 进入 `round_dispatch`。

分发 agent 读取：

- 主任务标题
- 主任务描述
- 原始请求内容
- 当前上下文
- 已注册 agent 和 tools
- 当前循环次数
- 最大循环次数

输出 `RoundPlan`。

如果 `RoundPlan.should_continue=true` 且存在子任务，则进入子任务执行。

### 4.6 子任务执行

每个子任务会先解析 agent：

1. 如果子任务里指定了 `assigned_agent_id`，优先使用该 agent。
2. 如果未指定或找不到，则用 mock 能力匹配逻辑按 agent capability 匹配。
3. 如果仍找不到 agent，则走 human mock 节点。

有 agent 的执行流程：

1. 模型读取主任务上下文、子任务、agent、已有工具结果。
2. 模型返回工具调用或最终输出。
3. 如果有工具调用，系统执行工具。
4. 系统把工具结果传回模型。
5. 模型生成最终输出。

无 agent 的执行流程：

- 当前使用 `mock_human_node_processing` 生成模拟人工处理结果。

### 4.7 上下文更新

每轮子任务执行完成后进入 `context_update`。

系统会：

- 合并本轮子任务输出。
- 生成 `TaskRound`。
- 写入 `context.rounds`。
- 更新 `context.summary`。

后续轮次的分发和执行都会读取最新上下文，因此后置子任务可以使用前置子任务结果。

### 4.8 完成判断

进入 `completion_judge` 后：

- 如果本轮计划表示无需继续，任务置为 `succeeded`。
- `final_output` 使用模型返回的 `final_output`，没有则使用 `context.summary`。
- 如果仍需继续，且未超过最大轮数，回到 `round_dispatch`。
- 如果超过最大轮数，进入 `human_intervention`。

### 4.9 Agent 轮询任务

agent 可通过 `POST /api/v1/agents/{agent_id}/poll` 查询分配给自己的任务。

当前轮询只是读取内存任务列表并过滤 `assigned_agent_id`，不是独立队列或消息系统。

## 5. 状态与事件

### 5.1 任务状态

`TaskStatus` 当前包含：

- `running`
- `succeeded`
- `failed`

### 5.2 当前节点

`CurrentNode` 当前包含：

- `intent_recognition`
- `human_confirmation`
- `waiting_dependencies`
- `dispatch_decision`
- `subtask_execution`
- `context_update`
- `agent_execution`
- `human_execution`
- `completion_judge`
- `human_intervention`

其中当前主流程主要使用：

`human_confirmation -> dispatch_decision -> subtask_execution -> context_update -> completion_judge`

必要时会进入：

- `waiting_dependencies`
- `human_intervention`

### 5.3 事件记录

任务会记录关键事件，例如：

- `task_created`
- `intent_recognized`
- `human_confirmed`
- `dispatch_decided`
- `agent_executed`
- `human_node_processed`
- `context_updated`
- `completion_judged`
- `human_intervention_required`

事件用于追踪任务生命周期，但当前不是完整审计日志。

## 6. 当前已验证行为

测试覆盖了以下行为：

- 创建 agent 并持久化到本地 JSON。
- agent 支持声明工具定义。
- 创建任务请求后进入人工确认节点。
- 意图识别会读取已注册 agent。
- 多个识别结果会合并成一个主任务。
- 确认任务后自动运行多轮流程直到成功。
- 没有匹配 agent 时走 human mock 节点并完成。
- 多轮任务会把上一轮上下文传给下一轮子任务。
- 工具执行器支持 mock 工具。
- 未注册工具会返回失败结果。
- 子任务模型可先返回工具调用，再基于工具结果生成最终输出。
- LangGraph 流程会把工具调用、工具结果和最终输出写入上下文。
- 模型客户端支持解析 Responses API、Chat Completions 等不同响应结构。

测试中真实模型调用通过 autouse fixture 禁用，避免单元测试依赖外部模型服务。

## 7. 当前边界与后续方向

### 7.1 当前边界

- Task 仅内存存储，服务重启会丢失。
- Agent 本地 JSON 存储无并发写保护。
- `execution_mode` 已记录，但当前子任务执行仍是同步循环，没有真正并发。
- `http` 工具能力较基础，没有鉴权、重试、审计和安全控制。
- 工具参数主要依赖模型输出和简单 URL 替换，缺少严格 schema 校验。
- human 节点目前是 mock 处理，不是真实人工任务队列。
- `agent_execution`、`human_execution` 等节点枚举存在，但当前 LangGraph 主流程没有直接使用这些节点。
- 完成判断主要依赖分发计划是否继续，模型完成度判断函数目前不是主路径。

### 7.2 后续方向

- 将 TaskStore 替换为数据库持久化。
- 为 AgentRegistry 增加并发安全和管理接口。
- 将 `execution_mode=parallel` 落为真正并发执行。
- 建立真实人工待办队列，替换 human mock。
- 增强工具执行安全策略，包括鉴权、超时、重试、白名单和审计。
- 为工具调用增加基于 `input_schema` 的参数校验。
- 增加任务事件审计表，区分业务事件和调试事件。
- 明确模型协议版本，减少 prompt 和解析逻辑耦合。
- 为模型失败、工具失败、部分成功增加更明确的状态流转。
