# 结构化任务持久化验证用例

## 用例名称

`test_database_storage_persists_structured_task_flow_tables`

## 用例文件

`tests/test_database_storage.py`

## 验证目标

验证一次完整任务流转后，系统不只保存 `tasks.payload`，还会按照结构化表设计同步持久化以下数据：

- 上游请求信息
- 主任务状态
- 任务执行轮次
- 派生子任务
- 任务事件轨迹
- 执行快照
- 工具调用结果

## 测试步骤

1. 使用临时 SQLite 数据库启动 FastAPI 测试应用，模拟数据库存储模式。
2. 调用 `POST /api/v1/agents` 创建带 `crm_query` mock 工具的处理 agent。
3. Mock 分发 agent，让它在第一轮生成一个 `Query CRM` 子任务。
4. Mock 处理 agent，让它先返回 `crm_query` 工具调用，再基于工具结果返回最终输出。
5. 调用 `POST /api/v1/tasks/requests` 创建业务系统任务请求。
6. 调用 `POST /api/v1/tasks/{task_id}/confirm` 进行人工确认，并触发后续自动任务流转。
7. 直接查询数据库结构化表，验证各表数据。

## 覆盖表

- `task_requests`
- `tasks`
- `task_rounds`
- `subtasks`
- `task_events`
- `task_snapshots`
- `tool_executions`

## 关键断言

- `task_requests.status` 最终更新为 `succeeded`。
- `task_requests.metadata_json` 保留上游 `external_id`。
- `tasks.status` 为 `succeeded`。
- `tasks.current_node` 为 `completion_judge`。
- `tasks.request_id` 与接口返回的 `request_id` 一致。
- `tasks.context_summary` 包含处理 agent 输出。
- `task_rounds` 记录第 1 轮分发原因和执行模式。
- `subtasks` 记录子任务状态、处理 agent、输出内容。
- `task_events` 记录完整事件顺序：
  - `task_created`
  - `intent_recognized`
  - `human_confirmed`
  - `dispatch_decided`
  - `agent_executed`
  - `context_updated`
  - `completion_judged`
  - `completion_judged`
- `task_snapshots` 至少包含：
  - `dispatch_output`
  - `subtask_execution_output`
  - `context_update`
- `tool_executions` 记录 `crm_query` 调用、成功状态和工具返回结果。

